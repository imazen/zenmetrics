# Per-codec picker training campaign — 2026-06-24

Trained rd-optimal config pickers on the 2026-06-24 Hetzner CPU corpus (ssim2 target, re-extracted 97
name@hex8 features). Objective: `size_optimal` (min bytes at a quality target). Pipeline: HistGB teacher →
MLP student → i8 ZNPR bake.

## Results — val mean overhead (how much bigger than rd-optimal the picked config's bytes are)

| codec | picker | val overhead | argmin acc | notes |
|---|---|---|---|---|
| webp | `zenwebp_picker_cpu2026.bin` (49 KB) | **1.20%** | 90.0% | 3-mode pick (vp8-m4/m6, vp8l) — easy |
| jpeg | `zenjpeg_picker_cpu2026.bin` (127 KB) | **5.87%** | 41.9% | 24 configs; dense 19-q halved coarse 10.25% |
| avif | `zenavif_picker_cpu2026.bin` (126 KB) | **9.34%** | 13.6% | 24 configs; intrinsically harder (flat win dist) |
| png | (no MLP — constant) | n/a | n/a | lossless: png-intense wins 99.7% → ship as default |
| jxl | **BLOCKED** | — | — | encode OOMs (butteraugli BufferPool, sibling bug) |

All bakes are i8 ZNPR, round-trip verified (max abs diff < 7e-5, n_inputs=204, n_outputs=24/3).

## Key findings

1. **Dense-q fixes jpeg, NOT avif.** jpeg coarse 7-q → 10.25%; dense 19-q → 5.87% (halved). avif coarse
   7-q → 9.18%; dense+coarse merge → 9.34% (no change). jpeg's bottleneck was q-resolution; avif's is
   **config-disambiguation** — its 24 speed×qm×chroma×bd configs are RD-close + content-dependent with a
   FLAT win distribution (top config 10.2% vs jpeg's 18.4%; speed s2 wins only 58.5%, s4/s6 win 41.5% so
   speed isn't collapsible). No dense-q or output-ablation lever helps avif; 9.34% is the realistic number.

2. **LOO > Spearman for feature ablation** (user directive). Spearman cleanup flagged 9 "redundant"
   features; dropping them HURT (5.87→6.07% + overfit). LOO (retrain dropping each, on a dedicated ccx63)
   showed the 0.999-Spearman "redundant" `noise_floor_y_p1` is the MOST valuable (+0.51pp). See
   `FEATURE_ABLATION_LOO_2026-06-24.md`.

3. **jpeg is robust to feature count.** LOO single-drops said 81/97 ablatable, but batch removal doesn't
   compose — overhead is ~5.9% from 28→97 features. No ablation lever (input OR output: all 24 configs
   are rd-used 0.6–18.4%); full-97 ships.

4. **Interpretable optimizer = the GBDT teacher** train_hybrid already trains (jpeg 6.38%, +0.5pp vs MLP),
   interpretable via feature importances + partial-dependence. Comprehensible decision drivers: target
   quality, image size, chroma sharpness, detail (laplacian variance), colour diversity.

## Deferred / blocked

- **jxl picker** — blocked on the jxl-encoder butteraugli `BufferPool` OOM (sibling crate; encode-path,
  config-independent; confirmed `anon-rss 15.5 GB` OOM at boot on a 15 GB box).
- **HDR picker** — HDR features extracted (1216 rows), but BLOCKED on HDR-encode infrastructure (verified
  2026-06-24): the `zenmetrics sweep` encode path REFUSES HDR PNG input — *"PNG signals an HDR transfer via
  cICP (transfer=16, PQ): refusing to crush it through the 8-bit SDR decode path — score it with --hdr
  instead"* (correct safety — pixels are sacred — but the SDR sweep therefore can't emit HDR avif variants).
  The sweep also has NO `--hdr` flag (HDR is `score-pairs --hdr` only), and cvvdp/iwssim are GPU-only in the
  current CPU build (no `cpu-cvvdp`). **To unblock:** (1) add an HDR-aware encode path to the sweep (decode
  HDR PNG → 10-bit avif-HDR) OR reuse the `datagen-2026-06-23-hdr` pipeline (which produced the jxl-HDR cvvdp
  sidecar) for avif; (2) score via `score-pairs --hdr --metric ssim2`/`zensim` (CPU, fast) or rebuild with
  `cpu-cvvdp` for the best-HDR-target cvvdp; (3) SPLIT-join → HDR Pareto → `train_hybrid --metric-column`.
  Features + corpus (`imazen-26-hdr-grid-2026-06-14`, 1140 renditions) are ready for steps 1–3.
- **output_bounds** — bakes emit open ± sentinels → runtime OOD-on-output is a no-op. train_hybrid should
  compute per-output (p01,p99) on val + emit `output_bounds`, then rebake.
- **Wiring into codecs (task A)** — pickers are artifacts; wiring into zenjpeg/zenwebp/zenavif is separate.
  Re-validate any rd_core ablation on the target corpus first (avif qm + per-axis ablations are
  corpus+metric+plan dependent — see `RD_ABLATION_2026-06-24.md`).
