# CUDA MPS context-init amortization — 2026-05-29

Task #142. Question: does **CUDA MPS** (Multi-Process Service) amortize
the ~181 ms per-process CUDA-context-init floor (#140) across separate
one-shot CLI processes? MPS runs a single shared CUDA context in a
daemon; client processes attach to it instead of each standing up their
own context. If that works, the second-and-later one-shot processes
should skip most of the ~181 ms floor, shifting the GPU↔CPU one-shot
crossover toward the GPU.

- **Data:** `benchmarks/gpu_mps_2026-05-29.tsv` (+ `.meta` provenance)
- **Host:** RTX 5070 (12 GiB), driver 596.21, **WSL2**
  (`6.6.114.1-microsoft-standard-WSL2`), CUDA SDK 13.2 + 12.6 installed
- **Baseline (MPS-off):** #140 cold-start measurements
  (`benchmarks/gpu_coldstart_2026-05-29.tsv`, git `0caf36d5`)

## The one-line answer

**MPS is unavailable on this host — it cannot be tested.** The MPS
control and server binaries (`nvidia-cuda-mps-control`,
`nvidia-cuda-mps-server`) do not exist anywhere on the filesystem. This
is a WSL2 limitation, not a missing-package fixable-in-place situation:
those binaries ship with the **Linux NVIDIA driver package**, and WSL2
has no Linux-side driver — the GPU is paravirtualized through the
Windows host driver. **No MPS-on numbers were measured (or fabricated);
the absence of the lever is the finding.** Answering the amortization
question requires a bare-metal Linux box.

## Feasibility check (the actual experiment)

The task's feasibility procedure was run verbatim. Every probe failed
the same way — no MPS daemon present:

| probe | result |
|---|---|
| `which nvidia-cuda-mps-control` | empty (not on PATH) |
| `find / -name 'nvidia-cuda-mps*'` (full filesystem) | **empty** — no control or server binary anywhere |
| `ls /usr/local/cuda*/bin \| grep mps` (all 5 toolkit installs) | empty |
| `dpkg -S nvidia-cuda-mps-control` | "no path found matching pattern" — no installed package owns it |
| `nvidia-cuda-mps-control -d` (start daemon) | `command not found`, **exit 127** |
| `echo get_default_active_thread_percentage \| nvidia-cuda-mps-control` | `command not found`, exit 127 |

What WSL2 *does* expose, in `/usr/lib/wsl/lib/`:
`libcuda.so.1` (user-mode CUDA driver), `libnvidia-ml.so.1` (NVML),
`nvidia-smi` — and no MPS daemon or server. (The `*mps*` hits under
`/usr/lib/wsl/drivers/` are unrelated Windows modem/printer INF files —
`mdmpsion.inf`, `MSMPS` print-pipeline — not CUDA MPS.)

GPU `Compute Mode` was `Default` before and after, and `nvidia-smi`
showed no running processes throughout — nothing was started, so there
was nothing to tear down beyond removing the two empty `/tmp/nvidia-mps*`
dirs the start procedure created.

## Why MPS is absent under WSL2 (root cause)

CUDA MPS is delivered as two host-side binaries —
`nvidia-cuda-mps-control` (the daemon/control utility) and
`nvidia-cuda-mps-server` (the per-user server hosting the shared
context). On a normal Linux box these come from the **NVIDIA display
driver package** (`nvidia-compute-utils-*` on apt, or the `.run`
installer), *not* from the CUDA toolkit — which is why all five
`cuda-*/bin` dirs here lack them despite a full SDK being installed.

Under WSL2 there is no Linux-side NVIDIA kernel/driver package at all.
The GPU is reached through `/dev/dxg` and the Windows host driver, with
only the user-mode CUDA driver shim mounted at `/usr/lib/wsl/lib`. The
MPS server's IPC/context-sharing path is not part of that paravirt
surface, so even installing the binaries (which aren't packaged here)
would not give a working daemon. MPS is a bare-metal-Linux feature.

the daemon up, process #1 pays the full floor (it triggers context
creation in the daemon) and processes #2–#N attach for a small fraction.
**That hypothesis is untestable on this host.**

## Implication for the GPU↔CPU crossover

The deployment lever MPS represents — amortizing the ~181 ms
context-init floor across independent one-shot CLI invocations — is
**not available on this WSL2 workstation.** So the #140 conclusion
stands unchanged here: a one-shot GPU score pays the full ~181 ms
context init (plus per-metric JIT and, for eager-alloc metrics,
allocation), and the one-shot crossover keeps CPU ahead for small
single images.

If MPS amortization matters for a target deployment, it must be both
**(a)** validated and **(b)** used on **bare-metal Linux** (or a Linux
container on a Linux host) with the full NVIDIA driver installed, the
MPS daemon running, and `Compute Mode = EXCLUSIVE_PROCESS`. On such a
host the expectation — to be measured there, not assumed — is that
process #1 pays the ~181 ms once and processes #2+ attach to the shared
context for a small fraction of it, which *would* shift the one-shot
crossover toward the GPU. **That measurement is deferred to a Linux
host; it cannot be produced here without fabricating numbers.**

## Caveats

- This is a feasibility result, not a performance measurement. No
  MPS-on timing exists because the daemon cannot run.
- A future bare-metal-Linux session can reuse the #140 drivers
  (`crates/<metric>-gpu/examples/coldstart_one.rs`, which time
  `Backend::client()`) unchanged: run them as fresh processes with the
  MPS daemon up (clients auto-attach via `CUDA_MPS_PIPE_DIRECTORY`) vs
  down, and compare `client_init_ms` for process #1 (cold daemon) vs
  #2–#N (warm daemon). The TSV schema in
  `benchmarks/gpu_mps_2026-05-29.tsv` already has the
  `mps_state[off|on-1st|on-Nth]` column ready for those rows.
