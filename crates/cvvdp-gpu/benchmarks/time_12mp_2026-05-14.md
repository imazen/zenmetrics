## Consecutive-weber diagnostic (3 runs)

```
run 0: weber#1=399.1 ms  weber#2=353.0 ms
run 1: weber#1=358.0 ms  weber#2=351.2 ms
run 2: weber#1=350.7 ms  weber#2=353.3 ms
```

The "weber(dist) is 2× weber(ref)" effect previously observed
inside `compute_dkl_d_bands` does not reproduce on consecutive
standalone calls — the cause is host memory pressure from the
`ref_weber` Vec held alive across the second call.

Tick 73's weber 21.6 ns/px was measured on a different day; the

machine-state (turbo / power / driver-level) noise needs a
proper paired sweep — the cuda perf is heavily sensitive to
warm/cool state.

## Open observations

- `d_bands` wall is dominated by 723 ms of work that is *not*
  the Weber pyramid. The next perf chunk is the per-band
  readback loop (`for k in 0..n_levels { client.read_one(...) }`)
  — that's `n_levels × 3 channels × band_size` round-trips,
  ~144 MB of GPU→host data at 12 MP.
- The 3-channel weber fusion (tick 89) is a launch-count win,
  not a wall-time win at 12 MP. The launch overhead is already
  amortized at this size; the fusion may matter more at smaller
  sizes (1 MP, 256 KP) where launch-overhead is a larger fraction.
