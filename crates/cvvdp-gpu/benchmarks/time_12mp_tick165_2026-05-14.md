# cvvdp-gpu — 12 MP timing snapshot (tick 165, 2026-05-14)

Run after tick 165 fused the 3 per-channel `pool_band_kernel`
launches per level into a single `pool_band_3ch_kernel` launch.
Cuts pool dispatch from `n_levels × N_CHANNELS = 24` launches
per JOD down to `n_levels = 8`.

Unlike tick 159's upscale 3ch fusion (which regressed by ~4%
because per-thread register pressure reduced warp-level
parallelism), the pool kernel does only 3 powfs + 3 atomic-adds
per thread — register footprint stays low and the fusion wins
purely on launch-overhead reduction.

## What changed

Added `pool_band_3ch_kernel` in `kernels/pool.rs`. Same per-pixel
math as `pool_band_kernel`:

```text
contribution = safe_pow(|v|, β) = (|v| + ε)^β − ε^β
partials[partial_idx].fetch_add(contribution)
```

The 3ch variant takes 3 input arrays and 3 partial-slot indices,
doing the atomic adds into 3 distinct slots per thread. Pool
atomics don't contend across channels (different slots), so the
atomic-throughput is the same — the win is purely launch-overhead
reduction.

In `compute_dkl_jod`, the per-level loop changes from:

```text
for c in 0..3 { pool_band_kernel::launch(d[k][c], partials, idx) }
```

to a single `pool_band_3ch_kernel::launch` per level. 24 → 8
launches per JOD at MAX_LEVELS=8.

New unit test in `tests/pool_scalar.rs`:
`pool_band_3ch_kernel_matches_per_channel_kernel` uses 3 distinct
input shapes and verifies each per-channel partial matches what
the single-channel kernel would have produced. Sanity asserts on
untouched partial slots catch a stray cross-channel-slot bug.

## Open observations

- The fusion delta (-7% jod) is consistent with what tick 159's
  upscale fusion expected but didn't deliver — the difference is
  per-thread work, not the fusion pattern itself. The decision
  rule for future fusions: if per-thread arithmetic is tiny
  (atomics, pointwise math), 3ch fusion wins; if per-thread
  arithmetic is medium (5-tap convolutions, multi-read patterns),
  3ch fusion loses to register pressure.
- The same pattern (very-low per-thread work + launch overhead
  dominant at small sizes) likely applies to `diff_abs_3ch_kernel`
  in baseband — that's already 3ch fused. Hard to find more
  candidates in this kernel class.
- Next targets: pool path is now near-minimal launch count;
  weber is still dominant. The remaining 6 upscale launches per
  weber level (3 channels × {v, h}) await a different angle
  than tick 159's naive fusion — shared-memory tiling that
  actually changes the memory access pattern.
