# cvvdp-gpu — 12 MP timing snapshot (tick 166, 2026-05-14)

Run after tick 166 eliminated the log_l_bkg readback + re-upload
roundtrip that `_dispatch_d_bands_into_scratch` was paying per
JOD. Adds a `log_l_bkg_dis` throwaway destination to
`WeberScratch` (parallel to tick 154's `bands_dis` split) so
DIST's log_l_bkg write doesn't clobber REF's data on
`weber_scratch[k].log_l_bkg`. The band loop's CSF kernel reads
from the GPU-resident handle directly — no host transfer.

## What changed

Tick 166 (`pipeline.rs`):

1. Added `WeberScratch.log_l_bkg_dis` — a throwaway destination
   handle parallel to `log_l_bkg`. Same shape (n_fine per
   non-baseband level). Allocated in `build_weber_scratch`.

2. `_dispatch_d_bands_into_scratch`:
   - REF weber dispatch writes to `weber_scratch[k].log_l_bkg`.
   - DIST weber dispatch writes to `weber_scratch[k].log_l_bkg_dis`
     (throwaway — per cvvdp's weber_g1 rule, both sides use REF's
     log_l_bkg, so DIST's is computed-then-discarded).
   - The ~64 MB readback loop over REF's log_l_bkg is gone.

3. Band-loop CSF kernel:
   - Non-baseband: reads `self.weber_scratch[k].log_l_bkg` GPU
     handle directly (no host upload).
   - Baseband: still does the small scalar-fill upload
     (`vec![log_l_bkg_baseband; baseband_n]`) since
     `weber_scratch[last].log_l_bkg` is uninitialized (baseband
     bypasses `subtract_weber_3ch_kernel`).

Bytes saved per JOD at 12 MP: ~128 MB (64 MB readback +
64 MB reupload). Sync drains per JOD: down by 7 (one per
non-baseband level was a sync read_one).

The `jod - d_bands` bucket shrunk from 92.6 ms (tick 165) to
23.9 ms (tick 166). That bucket includes both the pool
dispatch + the partials readback; the 64 MB savings show up
here because the partials readback used to wait for queued
pool kernels that themselves waited for queued log_l_bkg
uploads. Removing the upload-wait shortens the synchronous
critical path.

## Open observations

- 2×weber = 44.4 ns/px ≥ d_bands = 39.8 ns/px. The CSF + masking
  overhead is sub-noise; the band loop is bandwidth-tight
  against the 2 weber pyramid passes.
- The 5× single-thread / 2× 8-thread crossover is a meaningful
  threshold — the reference C+Zig implementation isn't trivially
  beatable. Future perf work has diminishing returns until a
  different approach (shared-memory tiling, multi-level fusion).
- Next target: weber pyramid itself remains dominant. The 6
  upscale launches per level (3 channels × {v, h}) plus the L_bkg
  expand pair are the largest remaining single-purpose chunk.
  A shared-memory tiled downscale could win bandwidth at 12 MP
  by reducing DRAM reads per output pixel from 25 to ~1.4 (256
  outputs share a 36×36 = 1296 src tile load).
