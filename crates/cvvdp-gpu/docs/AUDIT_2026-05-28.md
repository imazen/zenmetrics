# cvvdp-gpu doc audit — 2026-05-28

Cross-checks every documented claim about cvvdp-gpu's memory modes,
strip walkers, and kernel chunk status against shipped source.
Triggered by a downstream agent quoting "cvvdp-gpu Mode B at 4096² =
1505 MiB (-64% vs Full 4225 MiB)" — a number that doesn't appear in
the repo, yet whose **percentage is approximately correct** for the
current shipped state. The audit finds that STRIP_PROCESSING.md
stops documenting Mode B at the P2.1c landing (3457 MiB / -18.2%)
even though commits P2.3-P2.7 (2026-05-27) brought Mode B to
**1502 MiB / -64.4%** per the CHANGELOG.

## Summary

- 27 distinct memory/kernel/dispatch claims audited
- 18 verified accurate against source
- 9 stale / wrong / unverifiable, all in `STRIP_PROCESSING.md`
  (lines 281-365 — the "Phase 3 Approach B incremental landing" and
  "Next chunks" sections) and in the orchestrator `bench.rs:229-230`
  and `ORCHESTRATOR_DESIGN.md:130-132` design example

## A. Memory mode dispatch reality

## B. The "1505 MiB" provenance

**FINDING:** The exact string `1505` does not appear in source. The
closest documented number is **1502 MiB**, from `CHANGELOG.md:1312`
in the **P2.7 partial** landing entry (2026-05-27, commit hashes in
the same entry block):

> **Cumulative shrink from P2.1c baseline (4096² Mode B):** 3457 →
> 1502 MiB (-1955 MiB, -56.6%). JOD bit-identical at all tested
> sizes.

The downstream agent's "1505 MiB" appears to be a transcription error
of 1502 MiB (off by 3). The percentage **-64%** vs Full = 4225 MiB
checks out:

```
1502 / 4225 = 0.3556 → 35.5% of Full → -64.4% vs Full
1505 / 4225 = 0.3563 → 35.6% of Full → -64.4% vs Full (same to 1 sig fig)
```

So the downstream agent quoted ~accurate post-P2.7-partial numbers
but a slightly garbled MiB figure. The **bigger problem** is that
`STRIP_PROCESSING.md` documents the P2.1c-era numbers (3457 MiB /
-18.2%, at lines 1402-1414) as if they were the current state — the
five subsequent CHANGELOG entries (P2.3, P2.4, P2.5+P2.6, P2.7) did
not propagate into the doc.

That is the misbrief root cause: a session reading STRIP_PROCESSING.md
top-to-bottom finds the contradiction at lines 281-320 ("Memory
impact: zero so far. Only the pool stage iterates in strips") and
again at lines 1395-1418 (3457 MiB nvsmi for 4096² Mode B). Both
predate P2.3-P2.7. The CHANGELOG has the truth.

## D. `estimate_gpu_memory_bytes_strip_pair` semantics

**Function**: `pipeline.rs:1405-1547`. Models the **hybrid K_SPLIT
walker target state** (shallow levels strip-sized via
`mode_b_strip_h_at_level`, deep levels full-image), NOT the
currently-shipped state.

Components:
- Fixed: `src_bytes + srgb_lut_bytes + partials_bytes + logs_row_bytes`
- Pyramid: `3 × 3 × sum_level_pixels × 4` where `sum_level_pixels`
  counts strip-sized buffer pixels for `k < k_split` and full-image
  pixels for `k >= k_split`. `mode_b_strip_h_at_level` provides the
  back-projected strip height.
- `d_scratch`: persistent `d` (`3 × sum_level_pixels × 4`) + transient
  peak `(5 × 3 × largest_level_pixels × 4)` where `largest_level_pixels`
  is computed at level 0 strip size.
- `weber_scratch`: 6 fine + 4 vscratch sized at strip-aware pixels.
- `baseband_bytes`: full-image at deepest level.

The docstring claims it "models the hybrid K_SPLIT walker" — true
for the buffer allocation post-P2.4-P2.7. Reading
`build_d_bands_scratch:637-716`, `build_weber_scratch:459-635`, and
the `new_with_geometry_inner:2069-2192` allocations:

- `bands_dis_strip` allocated at `fine_w × R_k` per shallow level — ✓
  matches estimator's `sum_level_pixels`.
- `bands_ref_strip` allocated at `fine_w × R_k` per shallow level — ✓
- `weber_scratch.{l_bkg_fine, log_l_bkg, log_l_bkg_dis, vscratch_*}`
  shrunk to strip dims for shallow levels (P2.5+P2.6 — ✓).
- `DBandsTransient::new_strip` used in `_run_d_bands_strip_major_shallow`
  for shallow level transients (P2.4 — ✓).
- `gauss_ref` still full-image — NOT shrunk (P2.7 partial honest-stop
  per CHANGELOG:1296-1308). `gauss_alt` zero-sized for deep levels
  only; shallow levels still full-image.

**Worked example at `h_body=256, W=H=4096`:**
- Estimator predicts: **~835 MiB** (from
  `mode_b_walker_parity.rs:108-112` test output expectation
  `ratio < 0.25` of Full at 4096² — i.e. < ~1058 MiB out of 4225)
- Runtime measured per CHANGELOG:1311-1313: **1502 MiB nvsmi delta**
- Full nvsmi delta per CHANGELOG:1454: **4225 MiB**

The gap (1502 measured vs ~835 predicted) is `gauss_ref` + `gauss_alt`
still full-image at shallow levels (P2.7 partial honest-stopped).
The estimator is **optimistic** vs current runtime — it represents
the design target, not what allocates today.

**Recommendation**: add a clear prominent doc warning that the
estimator predicts the post-P2.7-full state, and that today's
runtime peaks higher by ~680 MiB at 4096² h_body=256.

## F. CappedPyramid behavior

`MemoryMode::CappedPyramid { levels }` at `memory_mode.rs:176`:

- Reduces the natural pyramid depth (`pyramid_levels` at
  `pipeline.rs` ~310) to the supplied `levels`, clamped against
  natural depth and against `MAX_LEVELS = 9`.
- Constructor: `Cvvdp::new_capped_pyramid` at `pipeline.rs:1949`,
  `Cvvdp::new_capped_pyramid_with_geometry` at `pipeline.rs:1973`.
- Estimator: `estimate_gpu_memory_bytes_capped` at `pipeline.rs:1251`
  — same accounting as Full but with the capped `n_levels`.

**Memory impact**: depends on how many deep levels get clipped. Per
the capped_pyramid_smoke.rs assertions, capping at `levels=5` for
4096² (natural depth ~9) saves enough to pass `capped5 + 1024 < full`
(very conservative). The doc claims "30-50% peak working set vs Full
at large images" (`memory_mode.rs:31`); this is consistent with
clipping `n_levels` from 9 to 5 — most pixel mass lives at levels
0-1, so big savings come from clipping pyramid OVERHEAD (d_scratch
persistent at deep levels, etc.). The number is plausible but the
doc cites the pre-rollback measurement, not a current bench.

**JOD impact**: doc claims `≤0.005 JOD parity gate at k=8 per
pre-rollback bench` (`memory_mode.rs:32`). The capped_pyramid_smoke
tests deliberately don't assert JOD parity — they accept up to JOD
10.001 (smoke level only). The 0.005 number is from a deleted
sweep — `archived/cvvdp_capped_levels_2026-05-22.csv` per
STRIP_PROCESSING.md:88-91 (no longer in the worktree). The number
is unverifiable without that file but it's a historical bench result,
not a current-state claim; it can stand as long as the doc says "per
pre-rollback bench".

**Auto resolver picks?** No — `resolve_auto` at `memory_mode.rs:197`
considers only `Full → Strip`, never `CappedPyramid`. Doc text at
`memory_mode.rs:30, 169` confirms "opt-in only".
