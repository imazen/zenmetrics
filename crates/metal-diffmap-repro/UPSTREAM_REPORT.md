# DRAFT upstream report — Metal (MSL) miscompiles dynamically-indexed storage read

> **Status: DRAFT for `gfx-rs/wgpu` (naga).** Do not post until the `[FILL ON
> METAL]` blanks are completed on a Metal machine and the bug is confirmed to
> reproduce by this standalone repro (not just the full zensim pipeline).
> Internal tracker: imazen/zenmetrics#20.

## Summary

A compute shader that is **correct on Vulkan (SPIR-V) and CUDA** produces wrong
results on **Apple Metal (MSL)**: a scattered subset of output elements come
back holding a **fixed value that is independent of the shader's inputs**, as if
a dynamically-indexed `read` from a `storage` buffer returns stale/uninitialized
memory instead of the written value. It only manifests once the buffers are
above a certain size (works at 64×64, fails at 96×80 and larger).

The WGSL is generated by [cubecl](https://github.com/tracel-ai/cubecl) (the
exact text is in [`generated_wgsl.txt`](./generated_wgsl.txt)), but the WGSL
itself is plain — a `%`, two `>>`, an `if/else` min-clamp, and a single
dynamically-indexed `storage` read — and it is correct everywhere except Metal.

## The suspect kernel (WGSL — correct, but mistranslated on Metal)

`pow2x_upsample_add_kernel` (nearest-neighbour upsample-add): each output `id`
computes a source index `sy_c * src_w + sx_c` from two right-shifts and a
min-clamp, then reads `buffer_0_global[that index]`. A single bad coarse element
gets NN-replicated into many output elements — which is exactly the observed
"many scattered outputs share one stale value" symptom.

```wgsl
@compute @workgroup_size(256, 1, 1)
fn pow2x_upsample_add_kernel(
    @builtin(global_invocation_id) global_id: vec3<u32>,
    @builtin(num_workgroups) num_workgroups: vec3<u32>,
) {
  let id = /* flat global index */ ;
  let l_0 = info.scalars_u32[2] * info.scalars_u32[3];   // dst_w * dst_h
  if id >= l_0 { return; }
  let l_2 = id % info.scalars_u32[2];                    // dx
  let l_3 = id / info.scalars_u32[2];                    // dy
  let l_4 = l_2 >> info.scalars_u32[4];                  // sx = dx >> log2_factor
  let l_5 = l_3 >> info.scalars_u32[4];                  // sy
  let l_6 = info.scalars_u32[0] - 1u;                    // src_w - 1
  let l_7 = info.scalars_u32[1] - 1u;                    // src_h - 1
  var l_mut_9: u32;  if l_4 < l_6 { l_mut_9 = l_4; } else { l_mut_9 = l_6; }  // sx_c
  var l_mut_11: u32; if l_5 < l_7 { l_mut_11 = l_5; } else { l_mut_11 = l_7; } // sy_c
  let l_13 = l_mut_11 * info.scalars_u32[0] + l_mut_9;   // sy_c*src_w + sx_c
  let l_14 = buffer_0_global[l_13];                      // <-- read returns stale on Metal?
  buffer_1_global[id] = buffer_1_global[id] + info.scalars_f32[0] * l_14;
}
```

(Full WGSL for all three kernels, including `per_scale_weighted_ssim_kernel`
which reads channel-offset indices `idx + pad_total*{1,2}`, is in
`generated_wgsl.txt`. If the upsample kernel turns out correct in isolation on
Metal, suspect that channel-offset read next.)

## What's ruled out
- Not our algorithm / not WGSL logic: identical WGSL is correct on Vulkan + CUDA.
- Not f32 precision: the wrong value is a *fixed* number, byte-identical across
  inputs that should change it — it's a wrong *read*, not arithmetic drift.
- Size-dependent (64 ok, ≥96 fails) and the bad outputs are NN-replications of a
  small number of coarse-buffer reads → consistent with one mistranslated
  dynamic `storage` index, not a global corruption.
