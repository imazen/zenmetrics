# Mode-wall sweep — Full vs Strip vs StripPair, all six metrics — 2026-05-31

Companion to `benchmarks/mode_wall_2026-05-31.csv` / `.meta`. Task #157
Phase A. Measures the WALL-TIME + peak-VRAM wall of each memory mode so
each metric's `resolve_auto` mode selection can be grounded in real
numbers. **MEASURE + recommend only — no resolver change (that is Phase
B, gated on a design decision).**

Hardware: water-cooled AMD Ryzen 9 7950X + NVIDIA RTX 5070 (12 GB), CUDA
13.2.1, driver 596.21. Harness commit `80deaa68`. GPU shared with another
process (~1.5 GB resident, ~10.3 GB free every cell — recorded per cell;
the contention flag never fired). n=20 timed reps/cell, wall p25/p50/p75,
strip body 256 rows. Subprocess-per-cell (cold cubecl pool each cell).

All 6 metrics × both contexts × 4 sizes are measured (96 cells; 8 cvvdp
strip/strippair cells are structurally invalid and recorded SKIP/FAIL —
see cvvdp section). The complete per-cell table is in
`mode_wall_2026-05-31.csv`.

## TL;DR — does "Full is fastest when it fits" hold?

**It depends entirely on the CONTEXT (one-off vs warm), and the current
resolver has no way to know the context.** Across every metric measured:

| context | small (256²–512²) | medium (1024²) | large (4096²) |
|---|---|---|---|
| **one-off** | Full fastest (butter: Strip) | crossover (≈tie) | **Strip / StripPair fastest by 1.7–15×, and 3–10× leaner VRAM** |
| **warm/batch** | Full fastest (iwssim: Strip) | Full fastest | **Full fastest** (strip 1.5–5× slower); strip leaner VRAM only |

Two metric-specific exceptions to the otherwise-uniform pattern: **butter**
prefers Strip in one-off even at small sizes; **iwssim** prefers Strip in
WARM at ≤512² (its BothStripped cached-ref beats Full's per-call IW recompute).

So:

1. **For the warm/batch path** (the orchestrator's cached-ref hot loop):
   "Full fastest when it fits" **HOLDS at every size**. Strip's only value
   is feasibility (running when Full OOMs) and a modest VRAM trim at 4K.
2. **For the one-off path** (`score_pair` / a single `compute_srgb_u8`):
   "Full fastest when it fits" is **WRONG at large sizes**. When the whole
   working set is constructed-and-thrown-away per call, the strip modes —
   which never allocate the full working set — win decisively on BOTH wall
   AND VRAM at 4096². At small sizes Full still wins (strip's per-strip
   dispatch overhead exceeds the construction it avoids).
3. **butter already deviates** — its `resolve_auto` is *strip-preferred*
   (picks Strip even when Full fits, for any image taller than the min
   strip height). The data says that is **right for one-off** (strip up to
   15× faster) but **wrong for warm** (Full 1.2–2.3× faster). So butter's
   current resolver is mis-tuned for the warm/batch path.
4. **No metric's `resolve_auto` ever considers StripPair.** For cvvdp,
   StripPair is the **fastest AND leanest one-off mode at 4096²** (−30 %
   wall, −43 % VRAM vs Full) — and it is unreachable through Auto today.

**Headline answer to the task's three questions:**
- *Does the heuristic hold?* Only in the warm context. In the one-off
  context it fails at ≥~2048² for every metric that has a strip path.
- *Is StripPair fastest for cvvdp one-off?* Yes at 4096² (−30 % wall,
  −43 % VRAM). No at ≤1024² (StripPair's constructor allocates more, so
  Full one-off is faster there).
- *Does Full's huge working set make strip faster at 4096² even when Full
  fits?* In the one-off context, emphatically yes (5–15× across metrics).
  In the warm context, no — Full's locality wins once construction is
  amortized.

**A correct data-grounded resolver NEEDS a one-off-vs-batch context hint.**
Its current signature `resolve_auto(width, height, cap)` cannot express
the single biggest factor in the answer. See "API-context question" below.

(MiB). `strip/full` and `spair/full` are p50 ratios (>1 = strip slower).

| warm | 256 | 2.87 | 3.21 | fail | 193 | 193 | — | Full |
| warm | 512 | 2.73 | 3.70 | fail | 225 | 226 | — | Full |
| warm | 1024 | 3.70 | 8.78 | fail | 385 | 386 | — | Full |
| warm | 4096 | 25.49 | 105.45 | fail | 3969 | 3970 | — | **Full** (Strip 4.1× slower, **same VRAM**) |

- **cvvdp Strip (Mode E) is cached-ref-ONLY** — a one-off `compute_srgb_u8`
  on a Strip-constructed cvvdp panics (`pipeline.rs:4963`, "ref_full_state
  must be Some"). Recorded as SKIP. There is no valid cvvdp one-off Strip.
- **cvvdp Strip warm gives NO VRAM benefit** (3970 vs 3969 MiB at 4096²)
  because Mode E keeps the full reference state on device. It is 1.1× (256²)
  → 4.1× (4096²) SLOWER than Full warm. **Mode E's only value is
  feasibility** (it can run when the dist-side Full working set would OOM);
  it is never a perf win.
- **cvvdp StripPair (Mode B) one-off** is the standout: at 4096² it is
  −30 % wall and −43 % VRAM vs Full, even though its compute "routes
  through Full" (`pipeline.rs:8407`: the per-strip walker isn't fully wired,
  but the StripPair *constructor* sizes the big transient buffers per-strip,
  which cuts the working set and the cubecl pool churn). At ≤1024² the
  StripPair constructor's extra setup makes it slower than Full one-off.
  This matches the earlier `cvvdp_mode_b_p21bc_2026-05-27` finding (Mode B
  faster than Full at 4096²).
- **cvvdp StripPair + warm FAILS at every size** ("image too small for the
  configured pyramid"). StripPair is a one-shot-pair mode with no cached
  ref of its own; warming it is a nonsensical combination and errors. This
  is a confusing error message (a clear "StripPair has no cached-ref; use
  Strip/Mode E for warm" would be better) but not a correctness bug in a
  valid path — flagged in the CSV as FAIL.
- **Parity (cvvdp warm, pure noise): |ΔJOD| ≤ 0.0032** (4096²), ≤ 1e-4 at
  ≤512². Within the documented Atomic<f32> reduction band → mode swap is
  JOD-safe for cvvdp.

## Parity — is a mode swap score-safe?

The task requires confirming each non-Full mode's score matches Full
within the metric's Atomic<f32> reduction-noise band. The main sweep uses
independent-noise inputs, which are **pathological for strip-halo parity**
(the per-crate `strip_parity.rs` tests deliberately avoid pure noise and
use gradient+checkerboard structure, asserting ~1e-4–5e-5 rel agreement).

Measured pure-noise parity (|mode − full|):
- **cvvdp** (warm): ≤ 0.0032 JOD — safe.
- **dssim**: bit-identical (0.000000) one-off AND warm — safe, even on noise.
- **ssim2** (warm): ≤ 0.009 on a ~−380 scale — bit-level, safe. (one-off
  strip diverges 0.6–4.2 — halo pathology, not a bug.)
- **butter**: diverges 1.5–2.1 on pure noise (halo pathology). The
  structured-input re-check (below) is the representative number.

**Structured-input parity re-check** (`MW_PARITY=1024` mode, gradient ref +
checkerboard dist, mag=12 — the ssim2/dssim strip_parity content shape),
|mode − full| at 1024²:

| metric | mode | \|Δ\| structured | scale | rel | verdict |
|---|---|---:|---|---:|---|
| cvvdp | strip (warm) | 0.000009 | ~5.8 JOD | 2e-6 | bit-level — **safe** |
| cvvdp | strippair (oneoff) | 0.0037 | ~5.8 JOD | 6e-4 | reduction-noise band — **safe** |
| dssim | strip (oneoff) | 0.000000 | ~0.27 | 0 | bit-identical — **safe** |
| iwssim | strip (oneoff) | 0.000000 | ~0.39 | 0 | bit-identical — **safe** |
| ssim2 | strip (oneoff) | 0.021 | ~−215 | 1e-4 | bit-level relative — **safe** |
| butter | strip (oneoff) | **1.31** | ~16.3 | **8e-2** | **DIVERGES ~8 % — see below** |

**Finding: butter Strip↔Full parity is CONTENT-DEPENDENT, the others are
robust.** Five of six metrics (cvvdp, dssim, iwssim, ssim2) agree to
≤1e-4 rel on structured content (and dssim/iwssim are bit-identical even on
pure noise) → a Full↔Strip(↔StripPair) swap is score-safe for them.

butter is different: it diverges ~8 % on the period-8 ±12 checkerboard.
**This is a content-frequency sensitivity, not a wholesale bug** —
butter's own `tests/strip_parity.rs` PASSES at 1e-4 rel (verified: 21/21
green, including 1024²/body-256, the exact config) on its `make_image`
content (smooth sinusoid + bounded `&0x3f` HF mix). The σ=3.22 HF-band
blur over an aggressive high-frequency checkerboard near a strip boundary
is the case where butter's finite-halo strip approximation drifts from the
whole-image result. On smooth + moderate-HF photographic content (what its
parity test models, and what production scores) butter Strip is bit-tight.

**Conclusion:** strip/strippair mode swaps are score-safe within each
metric's reduction-noise band on realistic content for cvvdp / ssim2 /
dssim / iwssim. **For butter specifically, a Full↔Strip swap can shift the
score several percent on aggressive high-frequency content** — so if Phase
B makes butter's mode context/size-conditioned, that swap is NOT
guaranteed score-identical on adversarial HF inputs (it is fine on the
smooth+moderate-HF content its strip_parity test covers). The large
pure-noise divergences in the MAIN sweep (butter 1.5–2.1, ssim2-one-off
0.6–4.2) are the adversarial-input worst case of the finite strip halo;
ssim2 collapses to bit-level on structured content, butter only partly.

## Recommended per-metric `resolve_auto` policy (for Phase B)

The fastest fits-the-cap mode by (metric, context, size):

| metric | one-off ≤512² | one-off 1024² | one-off 4096² | warm (all sizes) |
|---|---|---|---|---|
| cvvdp | Full | Full | **StripPair** | Full |
| butter | Strip | Strip | **Strip** | Full |
| ssim2 | Full | Strip≈Full | **Strip** | Full |
| dssim | Full | Full | **Strip** | Full |
| iwssim | Full | Full | **Strip** | Full ≥1024²; **Strip ≤512²** |
| zensim | Full | Full | Full | Full (only mode) |

iwssim is the one metric whose **warm** picks aren't uniformly Full:
its BothStripped cached-ref Strip is faster than Full warm at 256² (4.2×)
and 512² (1.5×), Full faster at 1024²+.

**Recommended policy shape:**

1. **Memory feasibility stays the hard gate (unchanged).** If Full doesn't
   fit the cap → Strip (or the largest fitting strip). This is what
   `resolve_auto` does today and it is correct. Never pick a mode that OOMs.

2. **When Full fits, the perf-optimal pick is context-dependent:**
   - **Warm/batch context → always Full** (every metric, every size).
     This is "Full fastest when it fits" and it holds. (For cvvdp,
     StripPair+warm is invalid anyway; Strip/Mode-E warm is strictly
     slower with no VRAM win.)
   - **One-off context → size-bucketed:**
     - ≤ ~1024²: Full (strip dispatch overhead dominates).
     - ≥ ~2048²: the strip mode (Strip for butter/ssim2/dssim; **StripPair
       for cvvdp**). 5–15× faster AND leaner — a free win on both axes.
     - butter is the exception that wants Strip from the smallest size in
       one-off.

3. **Add StripPair to cvvdp's Auto candidate set.** Today cvvdp `Auto`
   only resolves Full or Strip(Mode E); it never picks StripPair, which is
   the fastest+leanest one-off mode at 4096². (`ResolvedMode` would need a
   `StripPair` variant.)

4. **Re-tune butter's strip-preference to be context-aware.** Its current
   unconditional strip-preference is right for one-off, wrong for warm.

### Conservative alternative (if a context hint is NOT added)

If Phase B does **not** add a context hint to `resolve_auto`, the
safest single-mode default is **Full-when-it-fits for every metric**
(reverting butter's strip-preference). Rationale:
- The warm/batch path is the orchestrator's hot loop and the dominant
  production workload; Full is fastest there at every size.
- One-off large-image scoring is comparatively rare, and even there a
  Full that fits the cap is "only" 5–15× slower than the strip optimum
  but still completes — and the `MODE_SELECTION.md` budget explicitly
  tolerates this ("full can be 20pct slower and strip could be 3x slower
  to simply not fail"; feasibility > perf).
- This avoids shipping a resolver that is wrong for the common (warm) case
  in order to optimize the rare (one-off-large) case it cannot distinguish.

The strip win at one-off-4096² is large enough that it is worth capturing —
but **only** with a context hint, so the warm path is not penalized.

## API-context question (for the parent to decide Phase B)

**A correct data-grounded resolver needs a one-off-vs-batch context hint;
its current signature cannot express it.** Today:

```rust
pub fn resolve_auto(width: u32, height: u32, cap: usize) -> Result<ResolvedMode>
```

The single biggest determinant of the fastest mode is **context** (one-off
vs warm), which `(width, height, cap)` does not carry. The same `(w, h,
cap)` should resolve to **StripPair** for a one-off 4096² cvvdp but
**Full** for a warm 4096² cvvdp. A memory-only resolver literally cannot
make that call.

Two Phase-B options:

- **(A) Add a context hint to `resolve_auto`** — e.g.
  `resolve_auto(width, height, cap, ctx: ExecContext)` where `ExecContext
  ∈ {OneShot, Batch}`. This is a **public API signature change** to every
  per-crate `resolve_auto` (and the umbrella `MemoryMode::Auto` resolution
  path). Lowest-friction at the point of decision, but touches 6 crates'
  public surface.

- **(B) Leave `resolve_auto` memory-only; do perf-aware mode selection in
  the orchestrator chooser.** The orchestrator `Chooser`
  (`crates/zenmetrics-orchestrator/src/chooser.rs`) **already** takes an
  `ExecContext` (OneShot/Batch) and already ranks GPU backends
  (`GpuFull / GpuStrip / GpuStripPair`) by measured `ns_per_px` from a
  capability cache, picking the fastest that fits VRAM. It already has both
  the context AND the perf data that `resolve_auto` lacks. The cleanest
  design is: `resolve_auto` stays the pure memory-feasibility gate; the
  chooser owns perf-aware + context-aware mode selection, seeded from this
  CSV. No public `resolve_auto` signature change; the per-crate matrices in
  the chooser (`supported_backends`) gain the size/context-conditioned
  ranking.

**Recommendation: (B).** The perf-and-context-aware layer already exists in
the chooser, has the right inputs, and keeping `resolve_auto` as a
deterministic zero-cost memory gate matches `MODE_SELECTION.md`'s "never
measure perf inside `new()`" directive. Option (A) would duplicate the
context plumbing the chooser already has and force a 6-crate public API
break. The one concrete `resolve_auto` change worth making regardless is
adding **StripPair** to cvvdp's `ResolvedMode` candidate set so it is
reachable at all.

## Methodology notes / honest caveats

- **Wall numbers are valid regardless of input content** (timing/VRAM
  depend on size, not pixel values). Only the **parity** check is
  content-sensitive; the pure-noise parity divergences for butter/ssim2-
  one-off are an adversarial-halo artifact (see Parity), pending the
  structured-input re-check.
- **One-off context includes construction** inside the timed region (it is
  what a `score_pair` caller pays). This is why one-off p50s are far higher
  than warm p50s at the same size, and why strip's "no full-working-set
  alloc" advantage is so large in one-off at 4K.
- GPU was shared (~1.5 GB by another process); ~10.3 GB stayed free at
  every cell, so no VRAM peak was clipped and the contention flag never
  fired. The heaviest cell (ssim2 full 4096² one-off, 6.3 GB peak) fit
  cleanly.
- A few warm-context p50s show p75 > 1.4× p50 (e.g. dssim warm 512²); the
  cubecl pool / first-few-rep variance widens the tail. p25/p50 are the
  stable statistics; the medians are what the recommendation rests on.
