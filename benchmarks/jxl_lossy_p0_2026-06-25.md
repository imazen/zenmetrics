## CORRECTION: it's noise + DATA_STARVED, not clean convergence (90-img retrain)

Combined the 60-img local + 30-img fleet omnis (90 unique imgs, all imazen-26) and retrained:

| training data | student **val** overhead |
|---|---|
| 9 imgs (val 18 rows) | 39.2% |
| 60 imgs (val 136) | 17.66% |
| **90 imgs (val 185)** | **36.88%** |

**Non-monotonic** (39→17.66→**36.88**) — so the earlier "oracle gap converges with data" framing was
**over-reading two noisy points**. The honest reading: at 9–90 imgs the picker is `DATA_STARVED`
(84 (size,zq) cells <50 train rows in every run) and overfit (+31pp train→val here), so held-out
overhead is **high-variance (17–37%)**, not a clean downward trend. Adding the fleet's more-varied
renditions made the small-data fit *harder*, not easier.

**Implication for ≤1%:** the endpoint needs the **dense ML-discipline sampling** — enough rows *per
(size, zq, content) cell* (the gate's ≥50 threshold), via 20 log-spaced sizes × dense-q × k-means
representative images — not merely "more images." That is genuinely a fleet-scale dense-data
generation + multi-round iteration job. The pipeline + fleet are proven; the data protocol is the work.

## Finding: the monolithic chunk-sweep OOMs on medium/large imgs → job system needed for size diversity

Added `STRATIFY=1` to `hetzner_cpu_sweep.sh` (size-stratified selection: ~N/4 per tiny/small/medium/large
— the diagnosed fix for the size-skew). A 2-box stratified run (tiny=20/small=20/medium=20/large=17)
revealed the deeper blocker: the **tiny+small box completed, but the medium+large box was OOM-killed
even at SWEEP_JOBS=4 on a 32 GB cpx51** — the monolithic `zenmetrics sweep`'s cross-cell memory
high-water scales with **img-size × concurrency**, so larger imgs need ever-lower SWEEP_JOBS (slow +
flaky). This is the Known-Bug memory behavior at fleet scale.

**Implication:** the **size-diverse dense corpus that ≤1% needs cannot come from the monolithic
chunk-sweep** — it requires the **job system** (`zenfleet` per-cell, one encode per fresh process,
≤1.5 GB bounded), which is the doc's mandated vehicle precisely for this. That path needs the
real-plan-driven job-system launch wired (today `launch_fleet.sh` builds only a synthetic manifest;
the real flow is `--emit-cells` → `zenfleet-ctl declare-encodes` → manifest → workers). So the honest
critical path to ≤1% is: **wire the job-system real-sweep launch → size-stratified dense declare →
per-cell fleet run (memory-safe large imgs) → retrain → iterate**. The chunk launcher remains fine for
≤1 MP corpora (SWEEP_JOBS=4).

## Measured: chunk-sweep memory thresholds by image size (size-aware SWEEP_JOBS)

A medium/large-window run (SWEEP_JOBS=2, cpx51/32GB) pinned the thresholds:
- **tiny/small** (≤256²): SWEEP_JOBS=4 — fast, safe (RAM <10 GB).
- **medium** (~1 MP, e.g. 1024²): SWEEP_JOBS=2 — survives (sawtooth ~6 GB), but slow.
- **large** (>~1.4 MP, e.g. 933×1536): **OOM-killed even at SWEEP_JOBS=2** (sawtooth peak >32 GB on the
  heaviest cells). The monolithic sweep's per-image peak (encode + ssim2 + zensim + 372-feat extraction
  on a large frame) × even 2-way concurrency exceeds 32 GB.

**Refined conclusion (corrects the earlier blanket "job system needed for medium/large"):**
- The chunk-sweep covers **up to ~1 MP** with size-aware SWEEP_JOBS (4 small / 2 medium).
- **Large (>~1.4 MP) genuinely requires the job system** (one encode per fresh process, ≤1.5 GB) — the
  back-half (launcher real-manifest override + content-addressed corpus + jobexec) is mandatory for the
  large tier of the size-diverse dense corpus. Declare half already proven (13320 DesiredJobs).

So the next-session path is precise: small/medium tiers via chunk-mode (size-aware SWEEP_JOBS); large
tier via the job system; combine → size-balanced dense corpus → retrain → two grid-stable rounds.

## MEASURED root cause of the large-img OOM (disproves features/zensim/rayon guesses)

User flagged "measure don't guess; zensim should stream — maybe a conversion regression." Measured it
end to end (peak RSS, /usr/bin/time -v, local 32-core):

| probe | peak RSS |
|---|---|
| encode-only (mem_probe, e7 @1.33MP) | 83 MB |
| sweep big 1.33MP, `--metric ssim2`, no features, ~9 cells | 2013 MB |
| same `--metric dssim` / `--metric zensim` | 2024 / 2029 MB (**identical — not zensim-specific**) |
| `+ --feature-output` | +23 MB (**features are NOT the driver**) |
| small 128² (16K px), 9 cells | 57 MB |
| big, 3 cells vs 9 cells vs 60 cells | 405 → 2039 → 5621 MB (**grows with cell count**) |
| big 60 cells, RAYON_NUM_THREADS=1 / 2 / all | 5610 / 5644 / 5623 MB (**not rayon-threads**) |
| big 60 cells, `GLIBC_TUNABLES=mmap_threshold=128K` | 4073 MB (**−28%, partial**) |

**Conclusion:** the OOM is **allocator high-water that grows with (cells-per-process × image-size)** —
common to all metrics, single-threaded too, only ~28% recoverable via mmap-threshold. It is NOT a
zensim streaming/conversion regression, NOT the 372-feature extraction, NOT the encode, NOT per-core
rayon buffers. The earlier `NO_FEATURES` workaround was a misdiagnosis (kept as a valid option, but it
saves only ~23 MB).

**Fix:** bound cells-per-process. The **job system (one encode per fresh process)** resets the
high-water by construction — the robust fix (declare-half proven; back-half is the queued wiring).
`GLIBC_TUNABLES=mmap_threshold` added to the chunk worker as a free partial mitigation. Chunk-mode stays
viable for ≤1 MP at size-aware SWEEP_JOBS; large needs the per-cell job system.

### Tested the "stored reference image" hypothesis — bit-identical fix, measured RSS-neutral (NOT the driver)

User asked (verbatim) "is it a stored reference image?" and "Is it just the parallel mode?". The CPU
zensim `score()` path (`metrics/zensim.rs`) **does** rebuild the reference's full sRGB→XYB multi-scale
pyramid on **every cell** — confirmed in code — and zensim exposes an amortized API
(`Zensim::precompute_reference` + `compute_with_ref`, docstring "amortize across all encoder iterations")
the sweep doesn't use. So the hypothesis was mechanically correct. I implemented the fix (precompute the
reference once per image in `run`, thread `Option<&PrecomputedRef>` through `compute_cell` → reuse via
`score_with_precomputed`), gated behind a **bit-identity test** (`metrics::zensim::precomputed_matches_score`
+ a 36-cell integration diff: **0 mismatches**, scores byte-identical — pixel-sacred preserved).

Then measured a controlled A/B (two release binaries built from the same tree ± the change; identical
image / cells / flags; `/usr/bin/time -v` peak RSS; 1.05 MP `photo__medium__7040`):

| config | BEFORE peak RSS | AFTER peak RSS | wall Δ |
|---|---|---|---|
| `--jobs 1`, 36 cells | 3187 MB | 3181 MB (**−0.2%**) | −1.8% |
| `--jobs 1`, 144 cells (4 imgs) | 3194 MB | 3242 MB (**+1.5%**) | −1.5% |
| `--jobs 8`, 36 cells (mean of 3) | 3840 MB | 3770 MB (**within ±120 MB run noise**) | ~0% |

**Verdict — DROPPED the code (kept the finding):** the precompute-ref is correct + bit-identical but
**does not reduce peak RSS** (the held reference adds ~17 MB/MP resident, roughly offsetting the churn it
saves; it never lowers the per-cell transient peak the OOM is built from). It buys a marginal ~1.5–1.8 %
wall win **only at `--jobs 1`**, which **vanishes at `--jobs 8`** (production runs parallel; the serial
ref-build is parallelized across cores and hidden behind encode-saturation). Shipping it would add hot-path
signature complexity for zero production benefit, so it was reverted (recoverable via `jj op log`).

Answers to the three hypotheses, measured:
- **stored reference image?** — yes, rebuilt per cell, but fixing it (bit-identical) does **not** move RSS → not the driver.
- **just parallel mode?** — no. Parallel is higher (3.8 vs 3.0 GB: concurrent cells stack working sets) but serial *also* accumulates (jobs-1 grows 3→9→60 cells); both regimes, driver unchanged.
- **conversion regression / should stream-convert?** — zensim's per-call memory **is** bounded; the conversion isn't failing to stream. The driver is glibc retaining freed pages across many cells (high-water), dominated by per-cell **distorted-side + comparison + encode** allocations. The fix is bounding cells-per-process (job system), not changing the conversion.
